ItemsAdderAdditions resolves item and block references through one shared compatibility layer. The same ID formats are used by advanced recipes, advancement triggers, player predicates, and item-producing actions.

## Supported ID formats

| Content | Format | Example |
|---------|--------|---------|
| ItemsAdder item | `namespace:id` or bare `id` inside the current `info.namespace` | `my_pack:ruby_sword` |
| ItemsAdder block | `namespace:id` or bare `id` inside the current `info.namespace` | `my_pack:ruby_ore` |
| ItemsAdder furniture | `namespace:id` | `my_pack:ruby_lamp` |
| MMOItems item | `mmoitems:type:id` | `mmoitems:sword:flame_blade` |
| Vanilla item | `minecraft:id`, Bukkit material name, or bare Minecraft key | `minecraft:diamond`, `DIAMOND`, `diamond` |
| Vanilla block | `minecraft:id`, Bukkit material name, or bare Minecraft key | `minecraft:oak_log`, `OAK_LOG`, `oak_log` |
| Vanilla item/block tag | `#minecraft:tag` | `#minecraft:planks`, `#minecraft:logs` |

IDs are normalized to lowercase internally. For readability, prefer fully qualified IDs such as `minecraft:diamond` and `my_pack:ruby_sword`.

## MMOItems

MMOItems support is optional. If MMOItems is installed and enabled, use:

```yaml
item: mmoitems:sword:flame_blade
```

The plugin detects MMOItems stacks by their MMOItems type and ID. If MMOItems is not loaded, `mmoitems:type:id` references cannot be resolved and will not match.

## Vanilla tags

Tags are vanilla registry tags only. They can match vanilla items or vanilla blocks, but they do not match ItemsAdder content or MMOItems content.

```yaml
# Any vanilla plank item
item: "#minecraft:planks"

# Any vanilla log block
block: "#minecraft:logs"
```

Tags are supported in:

- `iaa_crafting_table` ingredients
- advancement item conditions and item predicates
- advancement block conditions and block predicates
- player predicate slots/equipment/location block checks

Tags are not valid as recipe results, campfire/stonecutter input items, furniture IDs, or action parameters that must create a concrete item stack such as `toast.icon`, `clear_item.item`, and `replace_item.item`.

## Rotated ItemsAdder blocks and furniture

When matching ItemsAdder blocks or furniture at runtime, directional suffixes are treated as the same base content:

```text
my_pack:chair_north -> my_pack:chair
my_pack:chair_south -> my_pack:chair
my_pack:chair_east  -> my_pack:chair
my_pack:chair_west  -> my_pack:chair
my_pack:chair_up    -> my_pack:chair
my_pack:chair_down  -> my_pack:chair
```

This makes advancement triggers and runtime checks work with rotated variants without having to list every direction.

## Recipe examples

```yaml
ingredients:
  P: "#minecraft:planks"
  M: "mmoitems:material:ruby_shard"
  I: "my_pack:reinforced_ingot"
result:
  item: "my_pack:ruby_handle"
```

```yaml
recipes:
  campfire_cooking:
    cooked_mmo_fish:
      ingredient:
        item: mmoitems:consumable:raw_fish
      result:
        item: mmoitems:consumable:cooked_fish
```

## Advancement examples

```yaml
criteria:
  chop_any_log:
    trigger: break_block
    conditions:
      block: "#minecraft:logs"
```

```yaml
criteria:
  obtain_custom_weapon:
    trigger: obtain_item
    conditions:
      items:
        - my_pack:ruby_sword
        - mmoitems:sword:flame_blade
        - minecraft:diamond_sword
```
